27.1 课程知识图谱回顾

本课程围绕生态学数据的生命周期展开,覆盖了从数据产生到知识发现的完整链条:

27.1.1 第一模块:基础篇(第一、二单元)

绪论与环境搭建和R 语言编程基础为整门课程奠定了基础。我们理解了数据采集和预处理在科学研究中的核心地位——没有高质量的数据,再精妙的分析方法也无法产出可靠的结论。同时,我们搭建了 R 和 Git 的工作环境,掌握了 R 语言的基本语法和 tidyverse 数据处理框架,学会了用 Git 管理代码版本。

关键概念:FAIR 数据原则、tidy data 理念、管道操作符 |>、Git 版本控制工作流

什么是 FAIR 数据原则?

FAIR 是四个英文单词的首字母缩写,代表了现代科学数据管理的核心理念:

  • F (Findable 可发现):数据应该容易被找到。这意味着你需要给数据分配唯一标识符(如 DOI),提供丰富的元数据描述,并将数据存储在可搜索的平台上。
  • A (Accessible 可访问):数据应该可以通过标准化的协议访问。即使数据有访问限制,访问的方式和条件也应该清晰说明。
  • I (Interoperable 可互操作):数据应该能与其他数据集、工具和工作流集成。这要求使用标准化的格式和术语。
  • R (Reusable 可重用):数据应该有清晰的使用许可,详细的元数据,以便他人可以在新的研究中重用。

为什么生态学研究需要 FAIR 原则?

生态学数据采集成本高、周期长。一个长期生态监测项目可能持续几十年,涉及多代研究人员。如果数据管理不规范,后来的研究者可能无法理解前人的数据,造成巨大浪费。FAIR 原则确保你的数据不仅对自己有用,也能为整个科学共同体贡献价值。

实践案例:如何让你的毕业论文数据符合 FAIR 原则?

假设你做了一个关于不同施肥处理对土壤微生物多样性影响的实验:

  1. Findable:将数据上传到 figshare 或 Zenodo,获得 DOI。在论文中引用这个 DOI。
  2. Accessible:使用 CSV 格式存储数据(而不是 Excel 的专有格式),任何人都可以用文本编辑器打开。
  3. Interoperable:变量命名使用标准术语(如”soil_pH”而不是”ph值”),单位明确标注(如”mg/kg”)。
  4. Reusable:提供数据字典(codebook),说明每个变量的含义、测量方法、单位。添加 CC-BY 许可,允许他人引用。

Tidy Data 理念深度解析

Hadley Wickham 提出的 tidy data 原则是现代数据分析的基石。它定义了数据应该如何组织:

  1. 每个变量是一列
  2. 每个观测是一行
  3. 每种观测单元是一张表

为什么这个原则如此重要?

因为大部分数据分析工具(包括 R 的 tidyverse、Python 的 pandas)都是基于这个假设设计的。如果你的数据符合 tidy data 原则,90% 的数据处理任务都会变得简单直接。

常见的”不整洁”数据形式

# 不整洁的数据:列名包含了数据值
messy_data <- data.frame(
  species = c("松树", "杉树"),
  plot_1 = c(120, 85),
  plot_2 = c(135, 92),
  plot_3 = c(128, 88)
)

# 整洁的数据:样地编号应该是一个变量
tidy_data <- data.frame(
  species = rep(c("松树", "杉树"), each = 3),
  plot = rep(1:3, times = 2),
  height = c(120, 135, 128, 85, 92, 88)
)

转换代码示例

library(tidyr)

# 将宽表转为长表
tidy_data <- messy_data |>
  pivot_longer(
    cols = starts_with("plot_"),  # 选择要转换的列
    names_to = "plot",             # 列名变成新变量"plot"
    values_to = "height",          # 数值变成新变量"height"
    names_prefix = "plot_"         # 去掉列名前缀
  )

# 查看结果
print(tidy_data)
#   species plot height
# 1 松树    1    120
# 2 松树    2    135
# 3 松树    3    128
# 4 杉树    1     85
# 5 杉树    2     92
# 6 杉树    3     88

为什么长表比宽表更好?

  • 易于分组统计:group_by(species) 就能按物种分组
  • 易于可视化:ggplot(aes(x = plot, y = height, color = species)) 直接画图
  • 易于添加新观测:新增一行即可,不需要添加新列

管道操作符 |> 的哲学

管道操作符让代码读起来像自然语言,从左到右、从上到下描述数据处理流程。

传统写法 vs 管道写法

# 传统嵌套写法(从内向外读,反人类)
result <- arrange(
  filter(
    select(data, species, height, diameter),
    height > 100
  ),
  desc(diameter)
)

# 管道写法(从上到下读,符合思维习惯)
result <- data |>
  select(species, height, diameter) |>  # 先选择需要的列
  filter(height > 100) |>               # 再筛选高度大于100的
  arrange(desc(diameter))               # 最后按直径降序排列

什么时候用 |>,什么时候用 %>%?

  • |> 是 R 4.1.0 引入的原生管道,速度更快,不需要加载额外的包
  • %>% 是 magrittr 包提供的管道,功能更丰富(如 . 占位符)
  • 建议:新项目用 |>,老项目保持一致即可

27.1.2 第二模块:数据采集篇(第三单元)

数据采集方法是课程的第一个核心模块。我们系统学习了生态学研究中的四大数据来源:

  • 野外观测:样方法、样线法、标记重捕法等经典生态学调查方法
  • 控制实验:实验设计原则、随机化、重复与对照
  • 模型模拟:用计算机模型生成和探索生态过程
  • 公开数据:GBIF、CNKI、气象站等公共数据资源的获取与使用

我们还学习了 Meta 分析的基本概念,以及如何用 Python 进行网络数据采集——这些技能让你能够突破单一数据来源的限制。

关键概念:采样设计(随机/系统/分层)、实验设计三原则(重复、随机、对照)、API 数据获取、网络爬虫伦理

采样设计的深度思考

为什么采样设计如此重要?

采样设计决定了你的研究结论能推广到多大范围。一个糟糕的采样设计,即使后期分析再精妙,也无法得出可靠的结论。

三种采样策略的适用场景

采样策略 适用场景 优点 缺点 生态学案例
简单随机 研究区域相对均质 无偏,易于统计推断 可能遗漏重要区域 均质草地的物种调查
系统采样 需要覆盖整个区域 空间分布均匀 可能与环境梯度共振 沿海拔梯度的植被调查
分层采样 研究区域异质性强 确保各类型都被采样 需要预先了解分层依据 不同林龄森林的土壤调查

常见错误1:样本量不足导致的统计功效问题

很多同学在设计实验时,只考虑”我能采多少样”,而不考虑”我需要采多少样才能检测到效应”。

错误示例

# 某同学设计了一个施肥实验
# 3个处理(对照、低氮、高氮),每个处理只有3个重复
treatment <- rep(c("对照", "低氮", "高氮"), each = 3)
biomass <- c(120, 125, 118,  # 对照
             135, 142, 138,  # 低氮
             155, 148, 152)  # 高氮

# 进行方差分析
model <- aov(biomass ~ treatment)
summary(model)

问题在哪?

每个处理只有3个重复,统计功效(power)很低。即使真实效应存在,也可能因为样本量不足而检测不到(犯第二类错误)。

正确做法:事先进行功效分析

library(pwr)

# 假设我们预期效应量为中等(Cohen's f = 0.25)
# 想要80%的把握检测到这个效应
# 显著性水平设为0.05
pwr.anova.test(
  k = 3,           # 3个组
  f = 0.25,        # 中等效应量
  sig.level = 0.05,
  power = 0.80
)
# 结果显示:每组至少需要52个样本!

实际建议

  • 如果资源有限,优先增加重复数而不是处理数
  • 每个处理至少5个重复,最好10个以上
  • 做预实验估计效应量和变异,再确定正式实验的样本量

常见错误2:伪重复(Pseudoreplication)

这是生态学实验设计中最常见也最致命的错误。

错误示例

某同学研究施肥对植物生长的影响,设置了2个大花盆:一个施肥,一个不施肥。每个花盆里种了10株植物,然后把这10株当作10个重复进行统计分析。

问题在哪?

真正的重复单元是花盆,不是植物。同一个花盆里的10株植物共享土壤、水分、微环境,它们不是独立的。这种情况下,真实的重复数是2,而不是10。

正确做法

# 错误的分析:把每株植物当作独立样本
wrong_data <- data.frame(
  treatment = rep(c("对照", "施肥"), each = 10),
  height = c(rnorm(10, 20, 2), rnorm(10, 25, 2))
)
t.test(height ~ treatment, data = wrong_data)  # p值会被严重低估!

# 正确的分析:先计算每个花盆的平均值
correct_data <- data.frame(
  treatment = c("对照", "施肥"),
  mean_height = c(mean(wrong_data$height[1:10]),
                  mean(wrong_data$height[11:20]))
)
# 但这样只有2个数据点,根本无法做统计检验!
# 正确的实验设计应该是:至少5个施肥花盆 + 5个对照花盆

实验设计三原则的实战应用

重复(Replication):为什么需要重复?

重复不是为了”凑数”,而是为了: 1. 估计实验误差(随机变异) 2. 提高统计功效 3. 使结论更可靠

随机化(Randomization):为什么必须随机?

随机化可以消除系统性偏差。假设你在温室里做实验,如果把所有对照组放在北侧、处理组放在南侧,光照差异就会成为混淆因素。

对照(Control):什么是好的对照?

  • 空白对照:完全不处理
  • 溶剂对照:如果处理组用了溶剂,对照组也要加等量溶剂
  • 阳性对照:已知有效的处理,用于验证实验系统是否正常

生态学案例:森林凋落物分解实验

研究问题:不同树种的凋落物分解速率是否不同?

27.1.3 第三模块:数据处理篇(第四单元)

数据清洗与质量控制教会了我们如何面对”脏数据”。现实中的数据总是不完美的——缺失值、异常值、格式不一致、重复记录。我们学会了用 tidyverse 工具链系统地处理这些问题,掌握了特征工程的基本方法,并建立了数据质量评估的框架。

关键概念:缺失值模式(MCAR/MAR/MNAR)、IQR 异常值检测、数据类型转换、特征工程、数据质量评估指标

缺失值的三种模式深度解析

理解缺失值的产生机制,是选择正确处理策略的前提。

MCAR (Missing Completely At Random 完全随机缺失)

缺失与任何变量都无关,纯粹是随机事件。

例子:你在野外调查时,有几个样方的数据记录纸被风吹走了。这种缺失与样方的植被类型、物种丰富度等都无关。

处理策略:直接删除缺失值不会引入偏差,但会损失信息。如果缺失比例<5%,删除问题不大。

MAR (Missing At Random 随机缺失)

缺失与其他已观测变量有关,但与缺失变量本身的真实值无关。

例子:你在测量树高时,特别高的树(>30m)因为测量困难,有些没有记录。但是否缺失只与”树高”这个变量有关,与其他变量(如胸径、树种)无关。

处理策略:可以用其他变量预测缺失值(如用胸径预测树高),多重插补是常用方法。

MNAR (Missing Not At Random 非随机缺失)

缺失与缺失变量本身的真实值有关。

例子:你在调查居民收入时,高收入者更倾向于拒绝回答。缺失本身就携带了信息(可能是高收入)。

处理策略:最棘手的情况。简单删除或插补都会引入偏差。需要建立缺失机制模型,或使用敏感性分析。

如何判断缺失模式?

library(naniar)
library(dplyr)

# 模拟数据
set.seed(42)
n <- 100
data <- data.frame(
  plot_id = 1:n,
  elevation = rnorm(n, 1000, 200),
  temperature = 20 - 0.006 * rnorm(n, 1000, 200) + rnorm(n, 0, 2),
  species_richness = rpois(n, 15)
)

# 检验缺失是否与其他变量相关
data |>
  mutate(temp_missing = is.na(temperature)) |>
  group_by(temp_missing) |>
  summarise(
    mean_elevation = mean(elevation),
    mean_richness = mean(species_richness, na.rm = TRUE)

异常值检测的多种方法

异常值可能是测量错误,也可能是重要的生态学发现。关键是要有判断依据。

方法1: IQR法(四分位距法)

# 计算异常值边界
detect_outliers_iqr <- function(x) {
  Q1 <- quantile(x, 0.25, na.rm = TRUE)
  Q3 <- quantile(x, 0.75, na.rm = TRUE)
  IQR <- Q3 - Q1

  lower_bound <- Q1 - 1.5 * IQR
  upper_bound <- Q3 + 1.5 * IQR

  outliers <- x < lower_bound | x > upper_bound
  return(list(
    outliers = outliers,
    lower = lower_bound,
    upper = upper_bound
  ))
}

# 示例:检测土壤pH异常值
soil_ph <- c(5.2, 5.5, 5.8, 5.6, 5.4, 5.7, 8.9, 5.3, 5.6, 5.5)
result <- detect_outliers_iqr(soil_ph)
print(paste("异常值:", soil_ph[result$outliers]))  # 8.9

方法2: Z-score法(标准化得分法)

适用于正态分布数据。

detect_outliers_zscore <- function(x, threshold = 3) {
  z_scores <- (x - mean(x, na.rm = TRUE)) / sd(x, na.rm = TRUE)
  outliers <- abs(z_scores) > threshold
  return(outliers)
}

# 示例
outliers_z <- detect_outliers_zscore(soil_ph)

方法3: 领域知识法

这是最重要的方法!统计方法只能告诉你”不寻常”,不能告诉你”错误”。

案例:土壤有机碳含量异常值判断

ggplot2 图形语法的深度理解

ggplot2 的核心思想是:一张图由多个图层叠加而成,每个图层都有明确的语法规则。

图形语法的七个组成部分

  1. 数据(Data):要可视化的数据框
  2. 映射(Mapping):变量如何映射到视觉属性(x, y, color, size等)
  3. 几何对象(Geom):用什么形状表示数据(点、线、柱等)
  4. 统计变换(Stat):对数据进行什么统计变换(计数、平滑等)
  5. 标度(Scale):如何将数据值转换为视觉值(颜色方案、坐标轴范围等)
  6. 坐标系(Coord):笛卡尔坐标、极坐标等
  7. 分面(Facet):如何将数据分割成多个子图

EDA 的目标是什么?

  1. 发现数据质量问题:缺失值、异常值、录入错误
  2. 理解数据分布:是否正态、是否有偏、变异程度
  3. 发现变量关系:相关、因果、交互作用
  4. 产生研究假设:为后续的确证性分析提供方向
  5. 选择合适的分析方法:根据数据特征选择参数/非参数方法

27.3.1 数据处理误区

误区1:直接删除缺失值

不分析缺失模式就删除,可能引入偏差。应先用可视化判断缺失类型(MCAR/MAR/MNAR),再选择合适的处理策略。

误区2:忽视异常值的领域含义

统计上的异常值可能是重要的生态学发现。先用领域知识判断,再决定保留还是处理。

误区3:在原始数据上直接修改

应始终保留原始数据不变,所有清洗操作写在代码中。

27.3.2 可视化误区

误区4:图表缺少必要元素

坐标轴标签、单位、图注是基本要求,不是可选项。科学图表必须能够独立于正文被理解。

误区5:颜色选择不当

避免红绿配色(色盲不友好),优先使用 viridis 或 Set2 等色盲友好调色板。

# 编写一个函数,检查数据框的质量问题
check_data_quality <- function(data) {
  # 1. 缺失值检查
  missing_summary <- data |>
    summarise(across(everything(), ~ sum(is.na(.)) / n() * 100)) |>
    pivot_longer(everything(), names_to = "variable", values_to = "missing_pct") |>
    filter(missing_pct > 0)

  # 2. 重复记录检查
  n_duplicates <- sum(duplicated(data))

  # 3. 数值型变量的异常值检查
  numeric_vars <- data |> select(where(is.numeric))
  outlier_summary <- numeric_vars |>
    summarise(across(everything(), ~ {
      Q1 <- quantile(., 0.25, na.rm = TRUE)
      Q3 <- quantile(., 0.75, na.rm = TRUE)
      IQR <- Q3 - Q1
      sum(. < (Q1 - 1.5 * IQR) | . > (Q3 + 1.5 * IQR), na.rm = TRUE)
    })) |>
    pivot_longer(everything(), names_to = "variable", values_to = "n_outliers") |>
    filter(n_outliers > 0)

  # 返回报告
  list(
    missing = missing_summary,
    duplicates = n_duplicates,
    outliers = outlier_summary
  )
}

27.9 常见问题速查表

27.9.1 R 报错速查 Top 10

  1. Error: object 'xxx' not found

    原因:对象不存在。检查是否拼写错误,或是否加载了对应的包(library())。

  2. Error in x[, "colname"] : incorrect number of dimensions

    原因:对象维度不对。用 str(x) 检查数据结构,可能需要 drop=FALSE 或改用 x$colname。

  3. Error: replacement has N rows, data has M

    原因:替换的向量长度不匹配。用 length() 核对。

  4. Error: cannot open file 'xxx.csv': No such file or directory

    原因:文件路径错误。用 file.choose() 交互选择文件确认路径,或用 getwd() 检查工作目录。

  5. Error: unexpected ')' in "..."

    原因:括号不匹配。检查是否少了括号或多了括号,用 RStudio 自动补全。

  6. Warning message: In x[i, j] : incorrect number of indexing results

    原因:索引结果为空。检查逻辑条件是否正确。

  7. Error: discrete value supplied to continuous scale

    原因:分类变量传给了连续型标度。改用 scale_fill_discrete() 或 scale_color_discrete()。

  8. Error: object 'filter' mask non-overlapping

    原因:dplyr::filter 与 stats::filter 冲突。用 dplyr::filter() 显式指定。

  9. Error: arguments imply differing number of rows

    原因:数据框列长度不一致。用 cbind() 或 length() 检查。

  10. Error: cannot allocate vector of size xxx Mb

    原因:内存不足。用 gc() 清理内存,或用 data.table 替代 data.frame 处理大数据。

27.10 科研可重复性专题

27.10.1 为什么可重复性很重要

2015年,Open Science Framework 复现了100项心理学研究,只有36项得到复现支持。这一”复现危机”揭示了一个问题:很多发表的研究结果是无法复现的。

生态学也面临类似问题。数据采集成本高、野外条件多变,使得可重复性研究在生态学中更为困难,但也更为重要。

可重复性的三个层次

  1. 计算可重复性(Computational Reproducibility):给定相同的数据和代码,能得到相同的结果
  2. 经验可重复性(Empirical Reproducibility):重复实验或调查,能得到相似的数据
  3. 统计可重复性(Statistical Reproducibility):重复分析,能得到相似的统计推断

本课程主要关注第一层次:计算可重复性。

27.10.3 预注册(Preregistration)的概念

预注册是在正式收集数据之前,在公开平台(如 OSF、AsPredicted)上注册你的研究计划(包括假设、方法、分析策略)。

预注册可以防止

  • HARKing(Hypothesizing After Results are Known):根据结果”倒推”假设
  • p-hacking:通过选择性报告得到显著结果
  • 确认偏误:只关注支持假设的数据

预注册步骤

  1. 在 OSF(https://osf.io)上创建账号
  2. 创建新项目,填写研究计划
  3. 明确写出:研究问题、假设、分析策略(哪些分析方法会使用)、判断标准
  4. 保存并获得时间戳(证明你在数据收集前就注册了)
  5. 数据收集和分析后,对比预注册内容和实际内容,说明任何偏离

注意:预注册不是”承诺书”,你仍然可以进行探索性分析。关键是要区分”确证性分析”(预先计划的)和”探索性分析”(数据驱动的)。

27.10.4 数据管理计划(DMP)的写法

数据管理计划说明你将如何管理整个项目的数据生命周期。对于国自然等课题申请,数据管理计划是必要的附件。

DMP 核心内容

  1. 数据类型:描述你将采集或使用的数据类型(数值、文本、图像、视频等)
  2. 数据采集规范:采样设计、记录格式、命名规则
  3. 元数据:每个变量代表什么、单位、取值范围、缺失值编码
  4. 存储与备份:存储位置、备份频率、备份介质
  5. 共享与隐私:数据共享时机、隐私保护措施
  6. 长期保存:数据保存期限、存储格式(推荐开放格式:CSV、TIFF、NetCDF)

27.11 数据分析方法论体系总结

不同类型的数据需要不同的处理策略。本课程涉及的数据分析方法可以按照以下框架组织:

27.11.1 按数据性质分类

  • 定量数据(数值型):描述统计 → 相关性分析 → 回归/方差分析 → 效应量计算
  • 定性数据(分类型):频次统计 → 卡方检验 → 对应分析
  • 计数数据(0,1,2,…):Poisson 回归 → 负二项回归
  • 比例数据(0-1之间):Beta 回归
  • 时空数据:插值 → 时空自相关 → 时空回归

27.11.2 生态学数据分析的特殊考量

空间自相关

相邻样点不独立,p值会被高估。

解决方案

  • 空间滤波器
  • 空间混合效应模型
  • 广义最小二乘法(GLS)
# 检测空间自相关
library(spdep)

# 创建空间权重矩阵
coords <- cbind(data$longitude, data$latitude)
nb <- knn2nb(knearneigh(coords, k = 4))
listw <- nb2listw(nb)

# Moran's I 检验
moran.test(data$biomass, listw)

生态学非平衡数据

很多生态数据不满足正态分布假设,优先考虑非参数方法或广义线性模型。

多变量协同

物种多度数据需要排序方法(PCA/CA/DCA)降维后再分析。

偶然性 vs 结构性

生态学格局往往同时受确定性和随机性过程影响,注意区分。

27.12.2 代码质量评分细则

优秀(90-100分)

  • 代码结构清晰,有注释说明
  • 使用管道操作符,逻辑流畅
  • 变量命名规范,易于理解
  • 包含错误处理
  • 可以一键运行,无需手动干预

良好(75-89分)

  • 代码基本规范,有部分注释
  • 逻辑清晰,但有改进空间
  • 变量命名基本合理
  • 可以运行,但可能需要小修改

及格(60-74分)

  • 代码可以运行,但结构混乱
  • 缺少注释,难以理解
  • 变量命名不规范
  • 需要较多手动干预

不及格(<60分)

  • 代码无法运行
  • 逻辑混乱,无法理解
  • 缺少必要步骤

27.13 结语

一学期的课程即将结束,但数据科学的学习才刚刚开始。

回想课程开始时,你可能对 R 语言一无所知,对 Git 感到困惑,面对一堆原始数据不知从何下手。现在,你已经能够独立完成从数据采集到可视化报告的完整流程。这种能力的提升,比任何单一的知识点都更有价值。

生态学正在经历一场数据驱动的变革。遥感技术、环境DNA、自动监测设备产生的数据量呈指数增长,能够熟练处理和分析数据的生态学家将越来越受到重视。你在这门课程中打下的基础,将在未来的学习和科研中持续发挥作用。

最后,记住几个原则

  1. 数据质量优先:再好的分析方法也无法弥补低质量的数据
  2. 可重复性至上:让你的分析过程可以被他人(包括未来的自己)复现
  3. 持续学习:技术工具在不断更新,保持学习的习惯比掌握某个具体工具更重要
  4. 动手实践:编程和数据分析是”手艺活”,只有不断练习才能真正掌握
  5. 批判性思维:不要盲目相信统计结果,始终用领域知识和逻辑推理检验结论

给未来的你的建议

  • 保持好奇心,对数据背后的故事保持敏感
  • 不要害怕犯错,每个bug都是学习的机会
  • 多看别人的代码,学习优秀的编程风格
  • 参与开源社区,分享你的经验和代码
  • 记录你的学习过程,写博客或笔记

祝你在未来的学习和科研道路上一切顺利!
